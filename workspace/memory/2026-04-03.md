## OpenClaw browser control incident + workaround

Problem:
- OpenClaw browser automation stopped working.
- Initial failure was a CLI/gateway protocol mismatch (`unknown method: browser.request`).
- After updating OpenClaw, the remaining failure was Chromium refusing to create `SingletonLock` under the managed OpenClaw browser profile path:
  - `/home/claw/.openclaw/browser/openclaw/user-data`
- Error persisted even after profile reset and enabling `browser.noSandbox`.

Resolution path taken:
1. Updated OpenClaw so CLI and gateway both reached `2026.4.2`.
2. Restarted gateway and confirmed protocol mismatch was gone.
3. Verified browser CLI auth/profile issue was caused by wrong config/profile invocation; corrected to use `/home/claw/.openclaw/openclaw.json`.
4. Confirmed managed profile still failed with:
   - `Failed to create ... SingletonLock: Permission denied (13)`
5. Verified Chromium works normally when launched manually against a temp user-data-dir under `/tmp`.
6. Tried fixing managed profile path directly:
   - reset/renamed profile dir
   - enabled `browser.noSandbox`
   - failure persisted
7. Determined managed OpenClaw profile path is the specific broken element on this host.
8. Worked around it by launching Chromium manually with:
   - user-data-dir: `/tmp/openclaw-browser-existing`
   - remote debugging port: `18950`
9. Added OpenClaw browser profile `tmp-cdp` using direct CDP websocket attach instead of managed profile storage.
10. Verified success:
   - OpenClaw browser profile `tmp-cdp` reports `running: true`
   - tabs list works
   - browser control restored through workaround profile

Important conclusions:
- Managed OpenClaw browser profile path is still broken on this host.
- Browser automation is restored via workaround profile `tmp-cdp`.
- Existing-session mode was not enough because it expected `DevToolsActivePort`; direct `cdpUrl` attach worked.

Current working browser details:
- Profile name: `tmp-cdp`
- CDP URL: `ws://127.0.0.1:18950/devtools/browser/13faa9a0-d663-4f04-b610-24e414aae636`
- Backing Chromium user-data-dir: `/tmp/openclaw-browser-existing`
- Manual launch pattern:
  `/usr/bin/chromium-browser --headless=new --remote-debugging-address=127.0.0.1 --remote-debugging-port=18950 --user-data-dir=/tmp/openclaw-browser-existing --no-first-run --no-default-browser-check --disable-dev-shm-usage --disable-gpu about:blank`

Follow-up recommended if revisiting root cause:
- Investigate why Chromium cannot create `SingletonLock` under `/home/claw/.openclaw/browser/openclaw/user-data` even though ownership appears correct.
- Likely buckets: confinement/AppArmor/snap behavior, hidden filesystem semantics, or Chromium expectations around that managed path.
